# 股票交易 Agent 项目第一阶段开发文档

## 1. 文档目标

本文档用于承接 `task001.md` 中的整体愿景，但只聚焦第一阶段最基础、最容易落地的部分，目标是先完成一个可以运行的最小可用版本（MVP）。

第一阶段只解决 4 个问题：

1. 用户有一个可用的前端聊天界面
2. 前端可以通过后端安全调用大模型 API
3. 系统可以保存基础会话数据
4. 系统为后续接入股票工具、记忆系统、多智能体预留清晰扩展点

一句话定义第一阶段：

> 先做出一个“能聊天、可扩展、能保存会话、后续能接股票工具”的基础系统骨架。

---

## 2. 第一阶段范围定义

## 2.1 本阶段必须完成

- GPT 风格基础聊天页面
- 会话列表
- 新建会话
- 发送消息并获得模型回复
- 后端代理调用大模型接口
- API Key 通过环境变量配置，不能硬编码到代码和文档中
- SQLite 持久化保存会话和消息
- 基础系统配置页或配置文件
- 预留股票工具调用接口
- 预留记忆系统接口
- 预留多模型适配层

## 2.2 本阶段明确不做

- 真正的多智能体协同
- 自动交易下单
- 复杂记忆检索
- Redis 短期记忆
- 向量数据库
- 股票多源聚合与复杂加权评分
- 用户权限系统
- 审批流、沙盒执行、契约引擎的完整实现
- 实盘交易连接券商

说明：

这些能力都很重要，但如果第一阶段一起做，项目会失控。当前原则是先建立稳定骨架，再逐步叠加能力。

---

## 3. 第一阶段产品目标

## 3.1 用户视角

用户进入系统后，至少可以完成以下操作：

1. 打开网页
2. 创建一个聊天会话
3. 输入问题，例如“分析一下宁德时代近期是否值得关注”
4. 后端调用大模型生成回答
5. 页面展示对话历史
6. 关闭页面后再次打开，历史会话仍然存在

## 3.2 工程视角

开发完成后，系统至少满足以下要求：

- 前后端分离，便于后续扩展
- 模型调用统一走后端，前端不暴露密钥
- 会话和消息结构化存储
- 模型层、股票工具层、记忆层相互解耦
- 能够在本地快速启动和调试

---

## 4. 第一阶段推荐技术方案

为了最快搭建稳定骨架，第一阶段不直接上完整多智能体框架，而采用更容易控制的基础栈。

## 4.1 前端

- `React`
- `TypeScript`
- `Vite`
- `Tailwind CSS`

选择原因：

- 开发快，生态成熟
- 很容易做出 GPT 风格聊天界面
- 后续接图表、行情卡片、表格都方便

## 4.2 后端

- `Python 3.11+`
- `FastAPI`
- `SQLAlchemy`
- `Pydantic`
- `Uvicorn`

选择原因：

- 与后续量化、数据抓取、股票分析生态兼容
- FastAPI 非常适合做 API 网关和工具编排层

## 4.3 数据存储

- `SQLite`

第一阶段只用 SQLite，原因：

- 零运维
- 足够支撑本地 MVP
- 后续可平滑升级到 PostgreSQL 或加 Redis

## 4.4 大模型接入

- 采用兼容 OpenAI 风格的后端适配层
- API Key 只保存在 `.env` 中
- 后端通过统一 `LLMProvider` 接口调用模型

说明：

你在 `task001.md` 中提供的密钥不能直接写入代码、文档或提交到仓库。开发时应改为环境变量方式，例如：

```bash
LLM_API_KEY=your_key_here
LLM_BASE_URL=https://your-model-provider.example.com/v1
LLM_MODEL=your-default-model
```

## 6. 第一阶段核心模块设计

## 6.1 前端模块

### A. 聊天主页面

功能：

- 展示消息列表
- 输入消息
- 发送消息
- 展示模型回复加载状态
- 展示错误提示

### B. 会话侧边栏

功能：

- 展示历史会话
- 新建会话
- 切换会话
- 删除会话（可选，若时间紧可延后）

### C. 顶部状态区

功能：

- 当前模型名称
- 后端连接状态
- 当前系统版本

### D. 股票扩展占位区

第一阶段先不做复杂股票分析，只做预留：

- “股票工具未接入”状态提示
- 后续用于显示行情卡片、新闻摘要、评分结果

## 6.2 后端模块

### A. Chat API

负责：

- 接收前端用户输入
- 调用 LLM 适配层
- 保存用户消息和模型回复
- 返回标准响应

### B. Session API

负责：

- 创建会话
- 获取会话列表
- 获取单个会话消息

### C. LLM Adapter

负责：

- 统一封装第三方模型调用
- 屏蔽不同供应商的接口差异
- 后续支持多模型切换

### D. Memory Stub

负责：

- 先定义接口，不做复杂实现
- 当前阶段仅返回最近若干条消息作为上下文

### E. Stock Tool Stub

负责：

- 先定义工具接口
- 当前阶段返回模拟数据或“暂未接入”
- 第二阶段再接真实行情和资讯源

## 7. 数据库最小模型设计

第一阶段建议至少有 3 张表。

## 7.1 sessions

字段建议：

- `id`
- `title`
- `created_at`
- `updated_at`

## 7.2 messages

字段建议：

- `id`
- `session_id`
- `role`，取值如 `user` / `assistant` / `system`
- `content`
- `created_at`

## 7.3 app_settings

字段建议：

- `id`
- `default_model`
- `temperature`
- `max_context_messages`
- `created_at`
- `updated_at`

说明：

第一阶段不需要提前设计太多交易表。会话和配置稳定后，再逐步补充股票、新闻、分析结果、观察池、持仓等业务表。

## 8. 最小接口设计

## 8.1 创建会话

`POST /api/sessions`

请求体：

```json
{
  "title": "新会话"
}
```

## 8.2 获取会话列表

`GET /api/sessions`

## 8.3 获取会话消息

`GET /api/sessions/{session_id}/messages`

## 8.4 发送聊天消息

`POST /api/chat`

请求体：

```json
{
  "session_id": "xxx",
  "message": "帮我分析一下贵州茅台",
  "model": "default"
}
```

返回体示例：

```json
{
  "session_id": "xxx",
  "reply": "以下是基于当前上下文的基础分析...",
  "tool_calls": [],
  "warnings": [
    "股票实时行情工具尚未接入，当前回答仅基于模型知识与已有上下文"
  ]
}
```

## 8.5 系统健康检查

`GET /api/health`

用于前端判断后端是否正常运行。

## 10. 开发步骤拆解

以下顺序必须严格执行，不要跳步。

## 第 1 步：初始化项目骨架

目标：

- 创建 `frontend` 和 `backend` 目录
- 初始化前后端项目
- 配置基础运行脚本

验收标准：

- 前端和后端都能本地启动

## 第 2 步：完成后端健康检查接口

目标：

- 实现 `GET /api/health`
- 前端可检测后端连通性

验收标准：

- 浏览器或接口工具可正确返回健康状态

## 第 3 步：完成会话管理接口

目标：

- 创建会话
- 获取会话列表
- 查询会话消息

验收标准：

- SQLite 中可看到会话与消息数据

## 第 4 步：完成基础聊天接口

目标：

- 前端发送消息到后端
- 后端调用 LLM
- 后端保存问答
- 前端渲染回复

验收标准：

- 用户在网页中输入问题后可以收到模型回答

## 第 5 步：完成 GPT 风格前端页面

目标：

- 聊天界面可用
- 会话切换可用
- 错误状态和加载状态可见

验收标准：

- 页面基础体验完整可用

## 第 6 步：加入基础配置管理

目标：

- 从环境变量读取模型配置
- 配置默认模型
- 配置上下文条数

验收标准：

- 无需修改源码即可切换基础模型参数

## 第 7 步：预留记忆与股票工具接口

目标：

- 定义 `MemoryService` 接口
- 定义 `StockToolService` 接口
- 在聊天链路中预留调用点

验收标准：

- 系统结构上支持后续接入，不需要推倒重来

## 13. 开发注意事项

## 13.1 安全要求

- 不要把 API Key 写入前端代码
- 不要把 API Key 提交到 Git
- 提供 `.env.example`，不要提交真实 `.env`

## 13.2 工程要求

- 先保证可运行，再逐步增强
- 每个模块单独可测
- 接口返回结构尽量稳定

## 13.3 产品要求

- 第一阶段不要追求“智能”
- 第一阶段先追求“稳定、可用、可扩展”

---

## 14. 最终结论

`task001.md` 是总蓝图，范围非常大；`task002.md` 定义的是第一阶段最小可执行方案。

当前最正确的起点不是立刻开发多智能体、记忆系统、股票推荐引擎，而是先完成以下闭环：

1. 前端聊天页面
2. 后端模型代理
3. 会话持久化
4. 未来扩展接口

只要这个闭环稳定，后续再接入记忆、股票工具、加权推荐、多智能体，项目才不会失控。
